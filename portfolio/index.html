Ancient Akkadians
https://hitchh1k3r.itch.io/ancient-akkadians

Another Ludum Dare game (LDJam 36) made by a 3 person team. I was the sole programmer, and once again pushed to do things that I was unfamiliar with. Most notably the game made use of a dynamic music system of my own design. The main track controlled the timing of all other tracks (Unity's playback system will allow audioclips to drift out of sync over long time periods), and then the volume of each track was set by the time of day system. Some tracks are crossfaded (ex. night crickets), and others change their volume immediately at a specific break point (used to turn music tracks on and off). Because some of the time rounds could be shorter than the musical phrases that were written for them, I had to implement a forward looking system, that would guess what time of day it would be half a musical frame ahead, and allow tracks to cut in early. Amorphous Space Widgets
https://hitchh1k3r.itch.io/amorphous-space-widgets

This was the only team Ludum Dare game I made with only one other person (they did sound and graphics). It had a lot of parts that got added as the project grew (not things that we had planned ahead of time). Implementing the game physics is the part that I remember most though. There were three shapes that all behaved differently. The triangle has three sticky corners that it can roll up a hill with. The square will squish a bit if something lands on it (it's a bit like a softbody), or it could wedge itself between two opposing wall (while falling) and slow down, eventually stopping. These mechanics were not translated into the level design though (which was rushed). All of the physics decisions were motivated by what would be fun to play, rather than what is realistic (I think a good choice for a 2D platformer).

Made for Ludum Dare 35, with the theme "Shapeshift". It was rated above average, probably because of a lack of cohesion.

Players were confused how to play, since implementing the mechanics took all of my time, I did not get a chance to build levels that taught the players very well. I added a tutorial level, and made a post-jam version a few days later, and I think it introduced the players to the game in a fun way, that was (in my opinion) better than the "game proper."

ASSET VARIABLES

A short description paragraph. Based on Ryan Hipple's Unite 2017 Talk. I made it a generic template, so any variable type can be easily created, and added a reverter system to make assetvariables act more like GameObject parameters than assets.

AssetVariable.cs

The AssetVariable and AssetReferance classes can be used for a lot of MonoBehavior properties. It is primarily a decoupling utility (preventing the pollution of the global scope with hard to qualify variables).

The scale parameters above are all FloatReferances, allowing the component user to provide a fixed value, or a reference to a FloatVariable (that can be set by an isolated script).


AUTO SAVE

AutoSaveSystem.cs

The auto save system will save the current opened scene at a fixed (configurable) interval. It live completely in the EditorAssebly, so it will not effect the build performance at all -- I'm not sure why so many Unity assets pollute the finished game with the scripts, when they really only change editor features. It is also smart enough to check if the file has already been saved (or if there are no changes since the last save), and avoid creating a new file.

ActionMap_StandardInspector.cs

My Input System allows the creation of various gameplay actions (things like "jump", "move", "take", or "open menu" -- standard actions types are: buttons, axes, 2d axes -- but the types are completely extendable, allowing a specific game type to provide it's own unique action types), and map them to similarly user defined (and extendable) inputs. Bellow you can see an example of a Keyboard & Mouse Action Map for various gameplay actions, in a fighting game.

The input assets currently wrap Unity's Input System, but they are fully extendable, so it would not be very hard to add new unique controller types. If you have the ability to get data from them, you can create an input map (and input types if the data cannot be represented an a button or axis). BUILD AND DEPLOY

A system designed to allow quick build and deployment (for team Game Jam environments). It follows a similar idea to my Input System, in that is is composed of separate extendable modules. The build part is fairly straight forward, and just triggers one of Unity's built in builds. The deployment part is composed of scripts that handle different cases. I've mostly used "WebsiteDeployment" which will zip the build (if it's an application, not if it's a web build), and then uses rsynch to upload it to a configurable server. There is a similar script for itch.io deployment (using butler), and no reason a Steam (or any other store) couldn't have support added very easily.

BuildAndDeployUtils.cs

Each deployment script can subscribe to several events (pre-build, post-build, build-error, etc.), and you can also make triggered by every build addons. Bellow is an example of a simple module that will write the current builds version number to a text file (useful for displaying the build version on the title screen).

VersionTextFile.cs

Each module can provided configuration options to be displayed (and stored) in a central place, for easy setup.

Builds can take a long time, it's nice to be able to chain builds and then take a break while they all process.
